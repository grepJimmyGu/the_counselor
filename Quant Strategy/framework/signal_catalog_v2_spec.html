Signal Library · v2 spec

Industry-Standard Semantics + Missing Primitives

Audit of the shipped PRD-16a signal catalog (~55 primitives) against the conventions retail and institutional quants actually use. Two failure modes uncovered: primitives over-rely on a single “value” output where the industry consumes events, regimes, and crossovers; and the catalog is missing entire computation idioms (proximity to extrema, squeeze detection, anchored references, ATR-trailing stops) that are table-stakes in modern quant workflows. This doc proposes the additive v2 fixes.

Two examples that motivate this work. (1) Today's MACD primitive emits the histogram’s sign — but the industry consumes MACD as a signal-line crossover event + histogram inflection + divergence pattern. (2) The SpaceX reference strategy used a “within 2–25% of the 52-week high” filter — that is not expressible with any primitive in the current catalog. v2 closes both gaps.

Section 1TL;DR — the two bugs

The shipped catalog from PRD-16a is structurally correct (categories, parameters, evidence tiers) but consumed wrong at the rule layer. Two distinct bug classes are conflated.

Bug 1 — Semantic flatness

Every primitive emits a single scalar value, and rules compare it to a threshold (RSI < 30, MACD > 0, ADX > 25). But the industry consumes most of these indicators as events (line A crosses line B), regimes (trending vs ranging classifier), or divergences (price vs indicator structure). Telling a user “buy when MACD > 0” is not the canonical MACD strategy — every textbook says “buy on bullish signal-line crossover.”

Bug 2 — Missing computations

Several common idioms are not expressible with current primitives: distance-to-extremum (“within X% of the 52-week high”), squeeze detection (Bollinger Bands inside Keltner), anchored VWAP from a catalyst date, ATR-trailing stops (Chandelier, Supertrend), smoothed candle direction (Heikin-Ashi), the academic 12-1 momentum z-score, and PEAD-style post-earnings drift. Each is industry-standard.

Section 2The semantics layer — six output kinds

Every primitive in the catalog can be classified by the shape of the signal it emits at each timestep. v1 implicitly assumed all primitives were “value” (a scalar to compare against a threshold). v2 makes the kind explicit and lets the composer UI render rule-builders specialized to each kind.

Output kindWhat the primitive emitsHow a rule consumes itExamples
VALUE A scalar at each bar (the raw indicator value) Compare against a numeric threshold rsi, atr, fcf_yield, macd_line
EVENT A boolean that is true only at the bar of the transition Triggers the rule on the bar it occurs; does not persist macd_signal_cross_up, price_breakout_52w_high, squeeze_fire
REGIME A categorical classifier of the current market state Acts as a gate / filter on other rules adx_regime (trending|ranging), vol_regime (high|low), squeeze_state
LEVEL A boolean that is true while a condition holds (persists) True for as long as the condition is met rsi_oversold (RSI < 30), price_above_200dma
DISTANCE A percentage gap between two reference values Compare against a range (between X% and Y%) distance_to_52w_high, distance_to_anchored_vwap
CROSS A specialization of EVENT: line A crosses line B Triggers on the bar of the cross; direction-aware ma_crossover_event, di_plus_above_di_minus, stoch_k_d_cross
DIVERGENCE A pattern: indicator’s swing structure disagrees with price’s swing structure Pattern-detection over a lookback window (peak/trough comparison) rsi_bullish_divergence, macd_bearish_divergence, obv_divergence

Why this matters for the composer UI

The composer (PRD-16b) currently renders one rule-builder shape: “[primitive] </> [threshold].” That is correct for VALUE primitives but inappropriate for the others. v2 lets the composer dispatch on output_kind:

Output kindRule-builder UI
VALUE[primitive] [< | > | =] [threshold] — same as today
EVENTWhen [primitive] fires — no threshold input (always-true=enable)
REGIMEWhen regime is [trending | ranging | …] — single-select
LEVELWhile [primitive] is true — no threshold input
DISTANCE[primitive] is between [min%] and [max%] — range input
CROSSWhen [line A] crosses [above | below] [line B] — direction picker
DIVERGENCEWhen [bullish | bearish] divergence detected over [N] bars

MACD family · Trend · Evidence A
Upgrade
Current state (v1)
One primitive macd with default_thresholds={"histogram_zero_cross": 0.0}. Output is the histogram scalar. A rule writes macd > 0 or macd < 0 as a bull/bear posture filter.
Industry pattern
MACD has three channels — MACD line (12-EMA − 26-EMA), signal line (9-EMA of MACD), histogram (MACD line − signal line). The canonical consumption (per StockCharts ChartSchool, Schwab, OANDA): (a) bullish/bearish signal-line crossover; (b) histogram inflection as an earlier signal than the crossover itself; (c) the zero-line position as a trend-bias filter; (d) bullish/bearish divergence with price. A pure “MACD line > 0” rule misses the canonical use case.
Recommended v2 primitives
Decompose MACD into one VALUE primitive emitting three channels, plus four event/divergence primitives that compose on top.

Section 4New primitives — complete computation reference

The full inventory of net-new primitives in v2 (i.e., not extensions of an existing family). Each has the formula in industry-standard form and the proposed schema entry for signal_primitives.py.

4.1 — TTM Squeeze (Bollinger ∩ Keltner)

Identifies low-volatility consolidation followed by a directional breakout. The most heavily used squeeze indicator in retail and institutional trading. Default parameters from John Carter's original specification.

# Inputs: 20-period MA, 2σ Bollinger, 1.5×ATR Keltner ma = close.rolling(20).mean() std = close.rolling(20).std() atr = atr_func(high, low, close, 20) bb_u = ma + 2.0 * std bb_l = ma - 2.0 * std kc_u = ma + 1.5 * atr kc_l = ma - 1.5 * atr squeeze_on = (bb_u < kc_u) & (bb_l > kc_l) # BB inside KC squeeze_fire = (~squeeze_on) & squeeze_on.shift(1) # transition off

4.2 — Supertrend

ATR-banded trailing trend line that flips when price closes through. Olivier Seban, 2009.

hl2 = (high + low) / 2 atr = atr_func(high, low, close, 10) upper = hl2 + 3.0 * atr # default mult=3 lower = hl2 - 3.0 * atr # Trend state ratchets: in uptrend, lower band can only rise. # Flip when close crosses the active band.

4.3 — Chandelier Exit

Volatility-adjusted trailing stop. Charles Le Beau, popularized by Alexander Elder.

# Long stop: 22-day HH − 3 × ATR(22) ce_long = high.rolling(22).max() - 3.0 * atr_func(high, low, close, 22) ce_short = low.rolling(22).min() + 3.0 * atr_func(high, low, close, 22) exit_long = close < ce_long

4.4 — Anchored VWAP

Running volume-weighted average price from a chosen anchor date (earnings, IPO, gap, macro event). Institutional reference for cost basis since the anchor.

# Slice bars from anchor_date forward slice_df = df.loc[anchor_date:] tp_vol = (slice_df.typical_price * slice_df.volume).cumsum() cum_vol = slice_df.volume.cumsum() anchored_vwap = tp_vol / cum_vol

4.5 — Relative Volume (RVOL)

Current volume ÷ average over a lookback window. The catalyst-confirmation primitive.

rvol = volume / volume.shift(1).rolling(20).mean() # Standard thresholds: 1.5 = elevated, 2.0 = surge, 3.0 = major catalyst

4.6 — 12-1 Momentum & Z-score

Academic equity momentum factor (Jegadeesh-Titman, MSCI methodology). 12-month return skipping the most recent month to avoid short-term reversal.

mom_12_1 = close.shift(21) / close.shift(252) - 1 # Cross-sectional z-score across universe at each timestep mom_12_1_z = (mom_12_1 - mom_12_1.groupby(level=0).mean()) \ / mom_12_1.groupby(level=0).std() mom_12_1_z = mom_12_1_z.clip(-3, 3) # Winsorize at ±3

4.7 — PEAD signal & drift window

Post-earnings announcement drift — Bernard-Thomas 60-day window. Use SUE (standardized unexpected earnings) as the signal magnitude.

# SUE = (actual - estimate) / std-error of historical estimates sue = (eps_actual - eps_estimate) / eps_estimate_std_err # Drift window: 60 trading days post-announcement days_since_earnings = bars_since(earnings_date) pead_drift_window = days_since_earnings <= 60 pead_signal = sue.where(pead_drift_window, 0)

4.8 — Heikin-Ashi smoothed candles

Noise-reduced candle structure. 35–45% less candle-to-candle variation per Technical Analysis Journal.

ha_close = (open_ + high + low + close) / 4 ha_open = (prev_ha_open + prev_ha_close) / 2 ha_high = max(high, ha_open, ha_close) ha_low = min(low, ha_open, ha_close) ha_trend = where(ha_close > ha_open, +1, -1) ha_streak = consecutive_count(ha_trend)

4.9 — 52-week extrema family

See Section 3 (52-week extrema family). Seven new primitives covering distance, breakout, ratio, sweet-spot zone, and recency.

4.10 — Z-score wrapper (utility)

Optional v2 enhancement: a meta-primitive that transforms any underlying VALUE primitive into a rolling z-score. Lets users compose “RSI z-score over 60 days” or “volume z-score over 20 days” without each primitive needing a hand-authored z-score variant.

# Generic: zscore_of(primitive_id, window=N) val = primitive(...) z = (val - val.rolling(window).mean()) / val.rolling(window).std()
Composition idea. The KB lookup endpoint (/api/signal-combos/match-templates) becomes meaningfully smarter once primitives are tagged by output_kind. A user dropping macd_signal_cross (CROSS) + rvol_surge (EVENT) onto the canvas can be matched to the “Breakout with volume confirmation” template family. Jaccard similarity on (category, output_kind) tuples beats Jaccard on category alone.

Section 5Schema implications (3 additive fields)

All three changes are additive. Existing primitives default to backward-compatible values. No saved strategy breaks.

5.1 — output_kind

The semantic-kind enum from Section 2.

class OutputKind(str, Enum): VALUE = "value" # scalar — existing default for all v1 primitives EVENT = "event" REGIME = "regime" LEVEL = "level" DISTANCE = "distance" CROSS = "cross" DIVERGENCE = "divergence" class SignalPrimitive(BaseModel): ... output_kind: OutputKind = OutputKind.VALUE # default keeps v1 untouched

5.2 — output_channels

Multi-output primitives (MACD, BBANDS, ADX, Stoch) expose multiple named channels a rule can reference.

output_channels: List[str] = ["value"] # default = single channel # Example: MACD SignalPrimitive( id="macd", output_kind=OutputKind.VALUE, output_channels=["macd_line", "signal_line", "histogram"], ... )

5.3 — composes

Derived primitives declare their parents. The composer uses this to render the input primitive’s parameter knobs and to wire the SignalProvider chain.

composes: List[str] = [] # default = standalone # Example: macd_signal_cross SignalPrimitive( id="macd_signal_cross", output_kind=OutputKind.CROSS, composes=["macd"], # inherits MACD's params; computes the cross event ... )
Composer-render dispatch. Once output_kind is on every primitive, <RuleBuilder> can route to a kind-specific sub-component (<ValueRule>, <EventRule>, <CrossRule>, …). The current single “value vs threshold” renderer becomes the default for VALUE only, and the rest get appropriate widgets. This is the highest-leverage downstream change from v2.

Section 7Migration plan

Three slices, each independently shippable. Each slice is a PR; each PR is backward-compatible.

Slice A — Schema fields (~3 days)

Add output_kind, output_channels, composes to SignalPrimitive. Backfill every existing primitive with the appropriate values (mostly OutputKind.VALUE with single-channel). Composer ignores the new fields. Zero behavior change.

Slice B — Audit upgrades (~2 weeks)

For each of the 11 anchor families, add the v2 primitives. Each primitive ships with its SignalProvider impl, a unit test, and a catalog entry. Existing rules using rsi < 30 continue to backtest identically.

Slice C — Composer rendering (~1 week)

Update <RuleBuilder> to dispatch on output_kind. EVENT, LEVEL, CROSS, REGIME, DISTANCE primitives render appropriate widgets (no threshold for EVENT/LEVEL; range input for DISTANCE; direction picker for CROSS). VALUE primitives unchanged.

Coordination with PRD-16c. PRD-16c's intraday extension multiplies the catalog: most v2 primitives are valid at both daily and intraday resolutions. Slice B should ship with resolution=["daily"] on all new primitives; PRD-16c flips eligible ones to ["daily", "intraday"]. Same pattern as v1.

What v2 explicitly does not do

  • It does not change any existing primitive's id, category, or provider_impl.
  • It does not add user-supplied custom Python signals — still a far-future Pro feature.
  • It does not add cross-asset signals (HY-IG spread, VIX term structure, put/call ratio) — those need new data_source wiring and are deferred to a separate PRD.
  • It does not redesign the KB-lookup endpoint — only enriches the Jaccard similarity input with output_kind.
