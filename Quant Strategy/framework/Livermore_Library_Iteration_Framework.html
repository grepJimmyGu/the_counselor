Livermore AI · Library Iteration Framework

A Repeatable Framework for Refreshing the Livermore Strategy Template Library

A quarterly cycle, scheduled-task-driven at the data-collection edges, human-reviewed at the decision points. Four steps — Collect · KB · Propose · Refresh — each with a single-file artifact, a definition of done, and a clear handoff. Backed by a markdown knowledge base that's the source of truth, mirrored into a Postgres index that Livermore queries at runtime.

Cadence: Quarterly Storage: Markdown + Postgres mirror Automation: Scheduled tasks + manual gate Effort per cycle: ~2 working days Library cap: 25 templates

1. North star & principles

The Livermore template library is a curated product, not an open marketplace. It exists to give users a small set of defensible, well-evidenced strategy starting points. The job of this framework is to make sure the library stays relevant over time without bloating or chasing fashion.

Defensible by default

Every production template has at least one Tier-A or Tier-B source. Tier-C is candidate-only, behind a Pro flag.

Small & opinionated

Hard cap of 25 templates in the gallery. Every new template forces a review of what to retire.

Markdown source-of-truth

You can read, diff, and rewrite the library with plain text tools. The DB is a mirror, not the truth.

Automation at the edges

Scheduled tasks do the boring work (telemetry pulls, raw web research). Humans do the strategic synthesis.

Cycle over cadence

If a cycle slips past 100 days, close it as-is and start the next. Better rhythm than perfect output.

Evidence over excitement

New papers must replicate or extend existing evidence. One-off backtest blog posts are research input, not promotion grounds.

2. The 4-step cycle

Each step produces exactly one markdown artifact in the cycle folder. Each step has a one-sentence definition of done.

2

3

4
Week 3–4

Step 1

Collect Livermore telemetry

Done when: every template in mvp/ and production/ has a verdict of keep / iterate / deprecate, with usage and feedback numbers cited.

What we measure

SignalWhy it mattersSource
Backtests per template (90d)Raw demand. Cold templates need a reason to stay.template_usage_events where event_type = backtest_run
Unique users per template (90d)Detects single-power-user noise.Same, group by user_id
Save rateDid the user think the result was worth keeping?strategy_saved events
Fork rateWas the template a good starting point for derivative work?strategy_forked events
Sandbox thumbs-down %Did the skeptical reviewer find something concerning?sandbox_thumbs_down events
Explainer thumbs-down %Was the explanation clear?explainer_thumbs_down events
Free-text feedbackConcrete language for next cycle's research.In-app feedback widget

Verdict rules

VerdictTrigger
keepbacktests_90d ≥ p50 AND thumbs_down_rate < 15%
iteratemoderate usage + high thumbs-down, OR low usage but introduced in last 1–2 cycles (still proving itself)
deprecatebacktests_90d < 5 AND template > 2 cycles old AND no recent positive feedback
If you don't yet have template_usage_events wired up in Livermore, treat your first cycle as "wire and observe" — ship the events first; the scorecard becomes meaningful in the second cycle.
Step 2

Knowledge-base scan & market regime view

Done when: at least one new KB entry under knowledge-base/ AND a 1-paragraph regime note in 02-research-scan.md.

Three parallel searches

  1. Academic — SSRN, arXiv-q-fin, NBER, Journal of Finance / RFS / JFE since the last cycle's end.
  2. Practitioner — AQR Insights, Robeco Insights, Two Sigma, GMO, Bridgewater letters from the last 90 days.
  3. Replication — replication or out-of-sample updates that challenge existing Tier-A evidence.

For each notable source

  1. cp knowledge-base/_kb-entry-template.md knowledge-base/<type>/<slug>.md
  2. Fill in the YAML front-matter (source_type, authors, year, tags, evidence_tier, strategies_referenced, dates).
  3. Fill in the body: TL;DR, Source Summary, Strategies Extracted, Quotes & Numbers, Cross-References.
  4. If this source supersedes an older one, set superseded_by in the older file.

Regime note — 1 paragraph

What regime are we in? What flipped recently? Which existing templates are likely hot vs cold in the next 90 days? This is the bridge to Step 3.

Step 3

Propose 2–3 new templates

Done when: 2–3 candidate spec files exist in templates/candidate/ using the _template-spec.md schema, with their rationale in 03-template-proposals.md.

The proposal funnel

Brainstorm wide — usually 6–12 candidate ideas from Steps 1+2. Score each on a 1–5 rubric across five dimensions:

DimensionQuestionScore 5 means
DemandDid Step 1 telemetry/feedback indicate users wanted this?Multiple explicit asks in feedback
EvidenceIs supporting research Tier A/B (not C)?Tier A with replication
FitDoes Livermore's current engine support this without major rework?Pure additive schema change
CapacityDoes it work at retail/prosumer scale?Works at $10k–$5M with no special data
DiffDoes Livermore already cover this with another template?Fills a real gap, no overlap

Top 2–3 by total score advance. Anything with Evidence ≤ 2 is auto-deferred.

Candidate file is born

For each chosen candidate, copy templates/_template-spec.md to templates/candidate/<slug>.md and fill the front-matter (slug, name, status: candidate, evidence_tier, capacity_tag, edge_type, horizon, asset_class, source_ids).

Step 4

Implement & refresh

Done when: new templates have coding-agent prompts written; promoted/deprecated templates have moved between lifecycle folders; the DB sync has been run; cycle-summary.md is filled.

Three parallel tracks

New templates

Write coding-agent prompts (one per template) following the v2 doc pattern: file paths, additive schema, engine branch, tests, acceptance.

Promotions

git mv from candidate/ → mvp/ → production/. Update front-matter + Promotion Log.

Deprecations

git mv to deprecated/. Set deprecation_reason and superseded_by. Open a Livermore ticket to hide the gallery card.

Sync to Livermore

# From repo root, after all `mv` commits
python apps/api/app/scripts/sync_knowledge_sources.py --plan    # dry run
python apps/api/app/scripts/sync_knowledge_sources.py --apply

python apps/api/app/scripts/sync_template_lifecycle.py --plan   # dry run
python apps/api/app/scripts/sync_template_lifecycle.py --apply

Close the cycle

Headline + 3 bullets in cycle-summary.md. Schedule the next kickoff (~80 days out). Move on.

9. Scheduled tasks (Cowork)

Two tasks set up once. Both are non-destructive — they only write to the cycle folders and never modify templates/ directly.

Task 1 — livermore-telemetry-monthly

Frequency: 1st of every month, 09:00 local. Cron: 0 9 1 * *.

What it does: runs the SQL queries against Livermore's DB, writes a CSV to /Quant Strategy/cycles/_telemetry-snapshots/<yyyymm>.csv, and raises a small flag file if any production template has backtests_90d < 5 or thumbs_down_rate > 25%.

Task 2 — livermore-quarterly-kickoff

Frequency: first Monday of Jan / Apr / Jul / Oct, 09:00 local.

What it does: clones cycles/_template/ to cycles/<next-cycle-id>/, pre-fills the scorecard in 01-collect-telemetry.md from the latest telemetry snapshot, runs a web search for top quant equity research in the last 90 days and dumps raw results to _research-raw.md, and pings the owner.

Full prompts for both tasks are at the bottom of framework/quarterly-runbook.md.

10. Template lifecycle

A template lives in exactly one of four folders at any time. Moving folders is the only way to change its stage.

candidate/proposed, not yet implemented in Livermore
mvp/implemented behind a feature flag or Pro tier
production/generally available in the gallery
deprecated/retired; file kept for audit

Promotion criteria

From → ToCriteria
candidate → mvpCoding-agent prompts shipped; backtest engine branch and tests merged green; default params smoke-tested.
mvp → productionLive for at least 1 full cycle; thumbs-down rate < 20%; at least 25 backtests run by ≥ 5 distinct users; no critical bug reports.
any → deprecatedVerdict deprecate from Step 1 for two consecutive cycles, OR Tier-A evidence is overturned, OR a clear superseding template is in production.

Deprecation hygiene

  • Set deprecation_reason (1–2 sentences).
  • Set superseded_by (the slug of the replacement, if one exists).
  • Open a Livermore ticket to hide the gallery card and surface an in-app notice to users with saved strategies of this type.
  • Do NOT delete the file. Audit trail matters.

11. Governance gates

Never skip these. They're the difference between a curated library and a graveyard of backtest noise.

Schema gate
Any new template must be expressible inside the current StrategyJSON schema, or the schema change has to be a separate, additive-only PR merged first (see the Strategy Library v2 doc Prompt 1).
Evidence gate
Tier-C templates ship behind a Pro flag with a visible "evidence: mixed" badge. Never in the Starter gallery. Tier-B requires at least one practitioner replication or a defensible engineering reason.
Capacity gate
Institutional-only strategies (heavy shorting, intraday-only, big-data-only) ship locked behind "requires data plan upgrade" with explanation. They never block on price; they block on data.
Cost gate
High-turnover templates default to slippage_bps ≥ 10. The UI warns if the user edits below this for those template types.
Bloat gate
Hard cap of 25 templates in the public gallery. New = retire one. No exceptions; if you can't pick something to retire, the new template isn't worth adding.
Stale-review gate
Any template not reviewed in 4 cycles auto-flags. The next cycle's Step 1 must address it.

12. Getting started checklist

One-time setup before the first scheduled-task cycle runs.

  1. Run the SQL in framework/sql-schema.sql against your Livermore database to create the two new tables, the events table, and the two views.
  2. Add lightweight event-logging to the existing Livermore routes:
    • POST /api/backtest → insert a backtest_run event keyed by template_slug.
    • POST /api/strategies (save) → strategy_saved event.
    • Sandbox / explainer thumbs → sandbox_thumbs_down / explainer_thumbs_down.
  3. Implement the three sync scripts in apps/api/app/scripts/ (see §8.4). Each is a small Python script that walks a directory, parses YAML front-matter, and upserts rows.
  4. Create the two Cowork scheduled tasks (§9). Test by running them manually first.
  5. Run cycle Q2-2026 (already seeded in cycles/Q2-2026/) end-to-end to validate the framework before depending on scheduled tasks for cycle Q3-2026.
  6. Add a 90-day calendar invite for the next cycle's kickoff.
You're now set up for a self-sustaining library. The framework's promise is modest: a small, defensible, evidence-grounded set of strategy templates that gets better every quarter and never bloats. Stick to the gates, trust the cadence.
