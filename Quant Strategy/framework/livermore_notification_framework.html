Livermore AI · Notification & Execution Framework

Once they adopt a strategy — when do we tell them, what do we tell them, how do we help them act?

Designed to mirror livermore_product_flow_v2's structure: a decision tree of six notification trigger states, per-trigger flows + modules, a maturity-level internal frame, channel IA, UI walkthrough, and engineering implications. Research-grounded — synthesizes Composer, Fidelity Trader+, Cboe Trade Alert, retail-platform alert-fatigue research, and Livermore's existing (partially-paused) Phase B signal infrastructure.

Positioning line. The platform's value compounds when users actually act on the signals we deliver. We win retention by making the path from "alert in inbox" to "trade placed" as short and as trustworthy as possible — and the way we lose users is by over-firing (alert fatigue) or under-explaining (no context = no trust).

Section 1Decision tree — six notification triggers

A signed-in user who has adopted a saved strategy can be notified by exactly six trigger types. The trigger is determined by what changed in the world, not by who they are. Each trigger has a default channel, a cadence ceiling, and a priority.

Design principle behind the ordering. Research (Cboe Trade Alert, Fidelity Trader+, retail-fatigue studies) is unanimous: real-time signal alerts drive immediate retention but cause churn if over-fired; daily digests are the highest-engagement, lowest-fatigue baseline. Ship P0 (Signal change + Daily digest) first; P1 (milestones + maintenance) deepens engagement; P2 (regime + education) is the long-tail coaching layer.

Section 2Per-trigger flows, info architecture & modules

For each trigger: the user flow, the information architecture (what fields the notification must contain to guide execution), the default channel(s), and the modules we need.

Trigger 1 — Signal change the action moment
P0 · Build first
"Your sector rotation strategy's monthly rebalance fired: sell XLK, sell XLY, buy XLU, buy XLP. Signal was based on close-of-business Oct 31 prices."

Product flow

  1. Cron computes signal end-of-day; detects a change vs cached state (existing signal_service).
  2. Writes a SignalEvent row (existing model).
  3. Dispatches to user's enabled channels (email default; in-app banner always; push if opted in).
  4. User clicks "View strategy" or "Mark as executed" from the notification.
  5. If "Mark as executed": Livermore stores the user's stated action timestamp (no broker integration — purely user-attestation).

Trigger 4 — Maintenance review the periodic check-in
P1 · Build next
"Your Sector Rotation hasn't rebalanced in 90 days — that's normal during trending regimes. Want to verify it's still working as designed?"

Trigger conditions

  • Strategy has been "in cash" for > 60 days (trend strategy that hasn't found a signal)
  • No rebalance in > 90 days for a rotation strategy
  • Performance lag vs benchmark exceeds threshold for > 3 months
  • Quarterly default: "It's been 90 days since you saved this strategy. Want to review?"

Information architecture

  • Plain-English reason for the quiet period (regime context)
  • Comparison: "Strategies like yours typically rebalance N times in this period"
  • Self-review prompts ("Has your goal changed?" "Do you still want this risk tier?")
  • Action options: "Looks fine, dismiss" / "Review parameters" / "Pause alerts"

Section 4Channel IA — the 4 channels + cadence rules

Four channels, each with a different latency / fatigue / cost profile. Every trigger has a primary channel and optional secondaries. The same notification never fires across multiple channels for the same event — fatigue prevention beats duplication.

📧 Email
Canonical. Audit trail. Always-on for all tiers.
  • Signal change (one per strategy per day max)
  • Daily digest (skipped on silent days)
  • Milestone
  • Maintenance (monthly throttle)
  • Compliance footer required
🪧 In-app banner
Visible on Home + My Strategies. No throttling.
  • Signal change (always shows if pending)
  • Milestone (until dismissed)
  • Maintenance (low-emphasis pill)
  • "Last signal: 3 days ago" status
  • Persistent until acknowledged
🔔 Web push
Opt-in real-time. PWA notifications. Pro+ tier.
  • Signal change only (others too noisy)
  • Title + 1 sentence body + tap-through
  • Browser-native; opt-in flow on first relevant trigger
  • Quiet hours (10pm–7am user local)
  • Throttled identically to email
🔌 Webhook / API
Pro-tier programmatic dispatch.
  • Signal change (real-time HTTP POST)
  • Milestone (optional)
  • JSON payload with full signal state
  • HMAC-signed; retry-on-fail (3 attempts)
  • Discord / Slack integration via standard webhook URL

Cadence + throttling — the load-bearing rules

The number one cause of unsubscribes is over-firing. Research across Cboe Trade Alert, Fidelity, LevelFields, and academic literature on alert fatigue agree: more than 1 email per strategy per day is the threshold beyond which open rates and trust collapse.
  • Per-strategy daily cap: max 1 signal-change email per strategy per day. If multiple rules fire for the same strategy on the same day, combine into one digest line.
  • Per-user daily cap (all email channels): 3. If exceeded, collapse into a single "Multiple alerts today — open digest" email.
  • Silent days for digest: if no signal changes overnight AND user opted "only when there's news", skip the digest.
  • Quiet hours for push: 10pm–7am user local; queue and deliver at 7am.
  • Snooze: 24h, 7d, "until next signal change" — single click from any notification.
  • Hard unsubscribe: one-click from any email; CAN-SPAM compliant; removes user from all email channels but preserves in-app and webhook.

What PR #107 paused — and what the re-shape should change. The original Phase B email dispatch (PR #88) fired one email per signal change with no per-strategy / per-user throttling and no daily-digest alternative. The re-shape should: (1) add the throttle layer up front, (2) ship the digest as the default channel rather than per-event email, (3) add the "Mark as executed" endpoint so we can measure the loop, (4) keep email + in-app banner as the only P0 channels (defer push + webhook to Phase C).

Compliance and trust guardrails (applies to every email)

  • "Not investment advice" footer on every email. Mandatory; legal-reviewed standard.
  • One-click CAN-SPAM unsubscribe in every email footer.
  • No "buy" or "sell" verbs in subject lines — use "signal", "rebalance", "rotation". Subject lines are public (preview in inbox); imperative verbs risk regulatory scrutiny.
  • Auditable provenance on every numeric claim — the backtest run + signal computation timestamp is stored, so we can re-derive any email's numbers months later.
  • No personalized recommendations — every alert is about the user's own saved strategy, not a Livermore-derived "we think you should…". The signal is the strategy's; we just deliver it.
  • "Mark as executed" is user-attested only, never a Livermore claim. We don't track whether the user actually traded; we track whether they self-reported acting on the signal.

The single retention metric to watch

"Time from notification sent → user clicks Mark as executed". If this metric is short and rising over time, the notification loop is working. If it's long or declining, either the alerts aren't trusted (information architecture problem) or aren't actionable (signal quality problem). It's the cleanest health signal we can instrument without integrating with brokers.
