Livermore AI · Product Flow v2 · Synthesized PM View

Six entry doors, four tabs, one product

A unified product framework synthesized from two prior drafts. Orders the six user entry modes by what retail users actually bring through the front door, maps them to a clean 4-tab navigation, and ends with a UI walkthrough showing what each user route looks like on the web. Decisions confirmed by Jimmy on 2026-05-26.

Positioning line. Livermore does not ask you to pick from dozens of strategies. It starts with what you already have — a ticker, a portfolio, a thesis — and guides you toward the testable strategy path that fits. The library is the backend inventory; the user experiences a guided journey, not a catalog.

Why this ordering changed from v1: retail users overwhelmingly arrive with a ticker, not a strategy concept. So "one asset" is now P0, ahead of the more abstract "I have an idea" flow which conflates two different mental models. The doc-synthesis call: split asset-first from idea-first; promote asset to P0.

What changed vs both prior drafts

  • "One Asset" promoted to P0. Most common retail starting point. Was conflated with "Idea" in the v1 HTML.
  • "Saved Strategy" removed from entry tree. Returning users land on Home (with a saved-strategies tile) or visit Strategy Builders directly — no separate tab needed.
  • "Custom Build" is a separate entry but visually de-emphasized for new users. Promoted as a primary CTA only after a user has done at least one template backtest.
  • Portfolio mode adopts the overlay paradigm wholesale (Section 2). This requires a backtest-engine extension — see Section 6.
  • Maturity levels are internal — used to decide which entry mode to surface most prominently for which user. Never shown as a survey to users.

Section 2Per-mode product flows & modules

Each mode has its own entry door but converges to the same backtest pipeline and the same saved-strategy lifecycle. Differences live at steps 1–2; steps 3–5 are shared infrastructure.

Mode 1 — One Asset most common retail entry
P0 · Build first
"I want to trade gold." · "What strategy fits NVDA?" · "Is oil better for breakout or mean reversion?"

Product flow

  1. Enter one ticker (or land via deep link from /stocks/[ticker]) Entry door: ticker search
  2. Asset classifier + behavior fingerprint "NVDA is a large-cap growth stock; trended 70% of the time over 10 years"
  3. Recommend 2–3 templates from the library with reasoning Wizard already does most of this
  4. Side-by-side comparison: return, drawdown, turnover, complexity Help user choose, not just pick top-1
  5. Pick template → 4-block summary → backtest → save / monitor Shared with all other modes

Key modules required

  • Ticker search + deep linkShipped
  • Asset classifier (stock / ETF / commodity)Partial
  • Asset behavior fingerprintMissing
  • Template recommender (5-question wizard)Shipped
  • Side-by-side comparison viewPartial
  • 4-block summary + risk presetShipped
  • Backtest + result viewerShipped
  • Save / signal alertsShipped
Reality: ~80% built. Behavior fingerprint (~3h) and a richer side-by-side comparison view are the missing pieces. Wizard already lands the user on this flow today.
Mode 2 — Portfolio retention engine
P0 · Build first
"I already own AAPL, NVDA, GLD, XLE, and SPY. What strategy should I use on this book?"

Product flow

  2. Portfolio diagnosis dashboard "Your book is 60% growth, 20% defensive, 20% commodity"
  3. Pick one of three overlay paradigms Not "replace your book" — overlay it
  4. Backtest the overlay (engine must accept inherited universe!) See Section 6 — engine extension required
  5. Save → monitor → email alert when overlay rule flips Standard lifecycle

Modules

  • Portfolio diagnosis (style / factor exposure)Missing
  • Backtest with inherited universeMissing (engine extension)
  • Multi-holding signal monitoringShipped
  • Email alerts per holdingShipped

Mode 3 — Thesis AI-native differentiation
P1 · Build next
"I think gold will rise when the dollar weakens and real yields fall."

Product flow

  1. Type thesis in plain English (text box, not chat) Entry door: thesis composer
  2. LLM parses into the thesis formula See formula below
  3. Show parsed structure + match to closest template "This is a macro-trend strategy on GLD"
  4. Optionally add filters (confirmation signals) "Dollar weakening, real yields falling"
  5. Backtest proxy strategy + regime counterfactual "In 2020-2022 environment this would have…"

The thesis formula

I believe [asset/universe] will move because of [driver], and I want to test it using [signal] with [entry], [exit], and [risk rule].

Modules

  • Thesis input UI (dedicated, not chat)Missing
  • LLM thesis parser → formulaPartial
  • Thesis-to-template matcherPartial
  • Confirmation-signal pickerMissing
  • Regime counterfactual analyzerMissing
Reality: chat builder covers ~30% today via strategy_builder_iterate. A dedicated "thesis as first-class object" UI is the actual unlock. ~1–2 weeks of work; most of it LLM prompt engineering.
Mode 4 — Custom Build prosumer / advanced
P1 · Build next
"I want to build my own strategy from scratch — or fork an existing one and modify it."

Product flow

  1. Choose starting point: template / thesis / blank Entry door: custom builder canvas
  2. Define WHAT (universe) Inherited or new
  3. Define WHEN IN (entry signal composer) Pick primitives + AND/OR — the missing piece
  4. Define HOW MUCH (sizing + risk preset) Already editable today
  5. Define WHEN OUT + optional confirmation filter Signal composer for exits too
  6. Backtest + review risk + save with version Shared with all other modes

  • 4-block canvas UIShipped
  • Template forking ("start from this template")Partial
  • Thesis-as-starting-pointMissing
  • Signal primitive catalog (frontend)Missing
  • WHEN IN / WHEN OUT composer (checkboxes + AND/OR)Missing
  • Optional confirmation filter UIMissing
  • Schema field logic_with_priorMissing

Reality: the 4-block summary UI exists today but WHEN IN / WHEN OUT are read-only template-baked copy. Closing this is the previously-documented "signal composition workstream" (~14h engineering). After that, Custom Build is real.
Mode 5 — Trading Idea concept-first user
P2 · Defer
"What about buying stocks after a 10% drop?" · "What if I bought oil after inventory drawdowns?"

Product flow

  1. Type idea or arrive via chat Entry door: chat (existing)
  2. Classify idea: trend / reversion / event / sentiment / macro "This is a short-term reversal idea"
  3. Check data availability "You'll need inventory data — that's macro tier"
  4. Match to closest template + fill missing rules "Want me to use RSI-MR as the proxy?"
  5. Backtest with risk warnings Sample size, overfitting, high turnover flags

Why deprioritize

  • Lower volume than asset-first or portfolio entries.
  • Existing chat builder via strategy_builder_iterate covers it ~60% today.
  • The risk-warning layer (overfitting, sparse data, high turnover) is the real unlock — but it benefits all modes, not just Idea.
  • Better to invest the warning layer effort cross-cutting rather than build a dedicated "Idea" entry.
Reality: route this through the chat builder as today. Don't build a dedicated Idea entry UI in v2.
Mode 6 — Discovery top-of-funnel browser
P2 · Defer
"I want to invest but don't know where to start." · "Show me what's hot."

Product flow

  1. Land on homepage / community feed Entry door: existing community surface
  2. See trending strategies + trending tickers Curated tile (Missing today)
  3. One-click "try it" with all defaults Anonymous backtest works
  4. Optionally sign up to save the result Conversion funnel handoff
  5. Converted user becomes Mode 1, 2, or 3 user Re-enters main funnel

Why deprioritize

  • Community feed + anonymous backtest already cover the basics.
  • A "trending strategies" tile is a 3–5 day add-on, not a workstream.
  • Don't invest in funnel-top features while the main funnel is leaking users (Modes 1 & 2 missing entry doors).
  • Build only after Modes 1 + 2 ship.

Section 4Navigation IA — keep the 4 existing tabs; trigger modes contextually

The product already has four top-level tabs. Entry modes are NOT new tabs. They are triggered by user actions within existing surfaces — the Home page, individual Stock pages, the Strategy Builders tab, and the Community page. This is a principle: more user routes ≠ more tabs.

Home
Main entry door; the "what do you have?" picker lives here
  • Replaces existing "Describe Your Strategy" section
  • Three entry CTAs: pick asset · upload portfolio · chat builder
  • Triggers Modes 1, 2, 3, 5 depending on click
  • For returning users: above-fold "Your saved strategies" tile
Market Pulse
Market state surface (existing); funnels into Mode 1
  • Top Movers, sector heatmap, macro snapshot
  • Each ticker links to its stock page
  • From stock page: "apply a strategy" button → Mode 1
Community
Discovery + thesis surface; funnels into Mode 3
  • Browse published strategies
  • Browse public theses
  • Click any thesis → opens Mode 3 thesis builder pre-filled
  • Fork any community strategy → opens Strategy Builders
Strategy Builders
Where Mode 4 (custom) lives, and where Mode 2 (portfolio) may be triggered from a template
  • Two paths: Use a template · Custom build
  • If a chosen template requires >1 ticker: "Use my portfolio" CTA → Mode 2
  • "Custom build" entry → Mode 4 four-block canvas
  • Houses the saved-strategies dashboard sub-section

Why this works. One of the doc's strongest principles is "library is inventory, journey is product." Adding a "Thesis Builder" tab would create a fifth surface with low traffic; embedding Mode 3 inside the chat builder on Home (and the thesis cards on Community) reuses surfaces users already visit.

Section 6Engineering implications & build queue

The v2 framing adds one substantial backend requirement and a handful of UI additions. Listed here in build order.

The big backend change: portfolio overlay engine

Engine extension required. The backtest engine today assumes the strategy defines its universe (e.g., a momentum strategy declares "S&P 500" as its universe). Portfolio mode inverts this: the user defines the universe (their holdings), the strategy is an overlay that inherits the universe and applies a rule on top.

What needs to change in BacktestEngine.run()

  • Add an inherited_universe parameter to StrategyJSON — when present, the strategy's rules apply to this universe instead of the strategy's default.
  • Three overlay types need to be added as new strategy_type literals:
    • portfolio_defensive_overlay — apply MA filter independently to each holding; cash out when each one breaks down.
    • portfolio_rotation_overlay — rank holdings by N-month momentum monthly; rebalance to top-K.
    • portfolio_rebalance_overlay — periodically re-weight back to original / specified allocation.
  • Engine emits per-holding signal instead of just per-strategy. So the live signal computation (signal_service) can email "your SPY rule flipped" rather than just "your overlay flipped".
  • Backtest result panel shows per-holding contribution to the overlay's P&L.

Estimated effort: 1–2 weeks of focused backend work. Most of it is the schema + engine; the three overlay strategy_types are straightforward once the inherited_universe abstraction is in.

What this does NOT include (consciously deferred)

  • Mode 5 (Trading Idea) dedicated UI — route through chat builder. Don't build a separate entry.
  • Broker connect for portfolio import — CSV / manual is fine for v2. Plaid / Snaptrade is a v3 investment.
  • Long/short engine, paper trading, options, walk-forward optimization — all institutional-tier, multi-week each.
  • User-facing maturity-level survey — the framing is internal-only.

The single thing to watch for

Don't fragment the backtest pipeline. All six entry modes (and especially the new portfolio-overlay engine extension) must funnel into the same BacktestEngine.run(), the same BacktestResult shape, and the same saved-strategy lifecycle. The differences live at the entry door (steps 1–2); the rest is shared infrastructure. If any mode spawns its own engine or its own result format, that's a flag — push back.
